**So what:** Build/pitch guidance: the thesis is defensible ONLY if Plumbline's verifier for these step types stays deterministic/structural or distilled-small, not a generic LLM judge. ## STRONGLY SUPPORTED — Intrinsic self-correction (a model fixing its own answer with no external signal) is unreliable, so Plumbline must use external/deterministic checks rather than just asking the model to check itself.

**So what:** Plumbline is right to build its verifier on EXTERNAL, ideally deterministic, signals (schema/tool-call validation, code execution, math recompute, citation-existence, cross-model disagreement, small independent judge/NLI models) and to NEVER rely on 're-ask the same model if it's sure.' The research even predicts the failure mode of a naive design: a self-doubt loop (FlipFlop) or a confirmatory reflection (>90% confirmatory) would flip correct steps to wrong and add cost without catching errors. For the YC/Datadog/Mistral judges, this is a strong, citeable defense of the architecture: frame Plumbline as 'the external signal a model cannot be for itself.' Two honest caveats to state on stage: self-correction DOES work when the step is externally checkable (so demo on verifiable step types — code, tool calls, structured output, math), and the verifier should be an INDEPENDENT model or deterministic check, not the generator grading itself.
